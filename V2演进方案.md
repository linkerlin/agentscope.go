## 1. 执行摘要

AgentScope Python v2（以下简称 PyV2）于 2026-05 正式发布，其本质并非功能堆砌，而是一次**架构范式转移**：从以 `Msg` 为核心的请求-响应模型，转向以 `AgentEvent` 为核心的流式状态机模型。这一变化影响 Agent 的每一个层面——从模型调用到工具执行，从人机协作到跨会话持久化。

AgentScope.Go（以下简称 Go 版）在 2026-04-15 完成了 ReMe Memory 深度整合及 Formatter/AgentBase/Hook 统一等 P0 补齐，已具备**独立生产级框架**的能力。但与 PyV2 对比，Go 版仍处于"消息为中心"的阶段，存在**一代架构代差**。

本方案确立 Go 版 V2 的核心目标：

> **在保持 Memory、并发性能、类型安全等 Go 原生优势的前提下，完成"事件驱动范式重构"，并补齐 Workspace、权限引擎、多租户服务等生产级能力，最终实现与 PyV2 的跨语言互操作能力。**

## 2. Python v2 范式转移解析

PyV2 的 rewrite 不是增量升级，而是以下四个维度的根本性重构：

### 2.1 通信原语：Msg → Event

| 维度 | v0.x / Go 版现状 | PyV2 |
|------|------------------|------|
| **核心原语** | `Msg` (消息) | `AgentEvent` (事件) |
| **Agent 输出** | `reply() -> Msg` | `reply_stream() -> AsyncGenerator[AgentEvent, None]` |
| **粒度** | 整条消息 | 块级增量 (`TextBlockDelta`, `ToolCallDelta`, `ThinkingBlockDelta`...) |
| **生命周期** | 无显式状态 | `PENDING → ASKING → ALLOWED → SUBMITTED → FINISHED` |

PyV2 定义了 **20+ 事件类型**，覆盖 Agent 运行的每一个原子操作：

```
ReplyStartEvent
├── TextBlockStartEvent → TextBlockDeltaEvent* → TextBlockEndEvent
├── ThinkingBlockStartEvent → ThinkingBlockDeltaEvent* → ThinkingBlockEndEvent
├── ToolCallStartEvent → ToolCallDeltaEvent* → ToolCallEndEvent
├── RequireUserConfirmEvent  ← HITL 挂起点
├── RequireExternalExecutionEvent  ← 外部执行挂起点
├── ToolResultStartEvent → ToolResultTextDeltaEvent* → ToolResultEndEvent
└── ReplyEndEvent
```

**关键洞察**：PyV2 的 UI、A2A、Observability 全部建立在这一事件流之上。没有细粒度事件，就无法实现真正的实时工具渲染、人机确认、跨 Agent 协作。

### 2.2 Agent 运行时：无状态 → 状态机

PyV2 的 `Agent` 是一个**可挂起、可恢复的协程状态机**：

- `AgentState` 持久化 `reply_id`, `cur_iter`, `cur_summary`, `context`, `tool_context`, `permission_context`
- `load_state()` / `save_state()` 支持跨 HTTP 请求恢复对话
- `RequireUserConfirmEvent` 使 Agent 在 `SUBMITTED` 与 `FINISHED` 之间显式挂起，等待外部事件驱动恢复

这意味着 PyV2 的 Agent 服务可以是无状态的 FastAPI worker，Agent 状态存储在 Redis 中。

### 2.3 工具执行：函数调用 → 中间件化沙箱

PyV2 引入了 `Workspace` 抽象和 `PermissionEngine`：

- **Workspace**: 隔离工具执行环境（Local / Docker / E2B）
- **PermissionEngine**: 规则引擎，支持 glob 路径匹配、命令子串匹配，决策 `ALLOW / DENY / ASK / PASSTHROUGH`
- **Toolkit Middleware**: 洋葱模式拦截 `on_reply / on_reasoning / on_acting / on_model_call / on_system_prompt`

工具不再是简单的函数调用，而是在**受控沙箱中经过权限校验、中间件处理、状态流转**的完整工作流。

### 2.4 部署形态：库 → 多租户服务

PyV2 的 `app/` 模块提供了完整的 FastAPI 服务：

- 多租户（用户隔离）
- 多会话（Session 管理）
- 多模型（Credential 管理）
- 定时任务（Scheduler / Cron）
- 存储抽象（`StorageBase` + `RedisStorage`）

这使得 AgentScope 从"开发框架"升级为"部署平台"。

### 3.2 关键滞后领域

| 领域 | 现状 | 与 PyV2 差距 |
|------|------|-------------|
| **事件系统** | 9 种粗粒度事件 | ❌ 缺少块级生命周期、缺少挂起/恢复事件 |
| **流式接口** | `CallStream` 是 `Call` 的包装 | ❌ 非真事件流，无法流式消费 tool result |
| **Agent 状态机** | `state` 包仅有空接口 | ❌ 无 `reply_id`、无挂起-恢复、无跨请求持久化 |
| **Workspace** | 完全缺失 | ❌ 无沙箱隔离、无 offloading |
| **权限引擎** | 完全缺失 | ❌ 无规则匹配、无 HITL 确认流程 |
| **Toolkit 中间件** | Registry/Group/Executor 基础分离 | ❌ 无洋葱中间件、无元工具、无异步 task_id |
| **MCP 深度集成** | ~755 行基础 client | ❌ 无 Server 适配、无动态 schema、无 stateful session |
| **多租户服务** | Gateway 为单机版 | ❌ 无用户/会话/credential 存储抽象、无 Redis 后端、无 Scheduler |
| **Context Compression** | Memory 层已具备 | ⚠️ Agent 层未自动集成（`buildHistory` 未调用 `PreReasoningPrepare`） |
| **ThinkingBlock/HintBlock** | `message` 包已定义 | ⚠️ Agent 循环和 Formatter 中未充分利用 |

## 5. V2 演进总体设计

### 5.1 设计原则

1. **范式优先**：第一阶段必须完成事件驱动范式重构，否则后续所有功能都会建立在错误的地基上。
2. **Go 地道性**：利用 Go 的 `interface`、`struct embedding`、`channel`、`goroutine`、`context.Context`，而非简单翻译 Python 的 asyncio。
3. **最小侵入**：保持 `Agent` interface 不变（或仅增加方法），`ReActAgent` 通过 embedding 复用新能力，现有用户代码不 break。
4. **优势放大**：Memory 系统继续领先，事件流与 Memory 的 `PreReasoningPrepare` 深度整合，形成差异化竞争力。
5. **跨语言互操作**：事件序列化格式、Msg JSON、A2A 协议与 PyV2 保持一致，确保 Studio UI 可以对接 Go 后端。

## 6. 第一阶段：事件驱动范式重构（P0）

> **目标**：完成从"消息为中心"到"事件为中心"的范式转移。这是 V2 的**地基工程**，所有后续功能依赖于此。
> **预估工期**：6-8 周

### 6.1 细粒度事件体系（P0-1）

#### 设计

在 `event/` 包（新建）中定义与 PyV2 对齐的事件层次结构：

```go
package event

// AgentEvent 是所有事件的根接口
type AgentEvent interface {
    EventType() string
    Timestamp() time.Time
    ReplyID() string // 关联到哪个 reply 会话
}

// 基础事件结构（struct embedding）
type baseEvent struct {
    eventType string
    timestamp time.Time
    replyID   string
}

// Block 类型常量
const (
    TypeReplyStart           = "reply_start"
    TypeReplyEnd             = "reply_end"
    TypeTextBlockStart       = "text_block_start"
    TypeTextBlockDelta       = "text_block_delta"
    TypeTextBlockEnd         = "text_block_end"
    TypeThinkingBlockStart   = "thinking_block_start"
    TypeThinkingBlockDelta   = "thinking_block_delta"
    TypeThinkingBlockEnd     = "thinking_block_end"
    TypeToolCallStart        = "tool_call_start"
    TypeToolCallDelta        = "tool_call_delta"
    TypeToolCallEnd          = "tool_call_end"
    TypeRequireUserConfirm   = "require_user_confirm"
    TypeRequireExternalExec  = "require_external_execution"
    TypeToolResultStart      = "tool_result_start"
    TypeToolResultTextDelta  = "tool_result_text_delta"
    TypeToolResultEnd        = "tool_result_end"
    TypeUserConfirmResult    = "user_confirm_result"
    TypeExternalExecResult   = "external_execution_result"
    TypeError                = "error"
    TypeInterrupt            = "interrupt"
)

// Delta 事件（携带增量内容）
type TextBlockDeltaEvent struct {
    baseEvent
    BlockIndex int
    Delta      string
}

// HITL 挂起事件
type RequireUserConfirmEvent struct {
    baseEvent
    ToolCalls    []*message.ToolCallBlock
    ConfirmID    string // 用于恢复时匹配
}

// 恢复事件（由外部注入）
type UserConfirmResultEvent struct {
    baseEvent
    ConfirmID   string
    Decisions   []ConfirmDecision // ALLOW / DENY / MODIFY
}

type ConfirmDecision struct {
    ToolCallID string
    Decision   string // "allow" | "deny" | "modify"
    ModifiedArgs map[string]any // 当 decision = modify
}
```

#### 与 PyV2 对齐要点

- **序列化格式**：JSON 字段名与 PyV2 完全一致，确保跨语言事件流可解析。
- **ReplyID**：每个 `reply_stream()` 调用生成 UUID，所有子事件携带该 ID，便于追踪和状态恢复。
- **BlockIndex**：同一 reply 内可能存在多个 `TextBlock` 或 `ToolCallBlock`，用 index 区分。

### 6.2 真事件流 `CallStream`（P0-2）

#### 现状问题

当前 Go 版的 `CallStream` 实现逻辑近似于：

```go
func (a *ReActAgent) CallStream(ctx, msg) <-chan *message.Msg {
    ch := make(chan *message.Msg)
    go func() {
        defer close(ch)
        result, err := a.Call(ctx, msg) // 仍然是先完整执行 Call
        if err != nil { ... }
        ch <- result
    }()
    return ch
}
```

这不是"流式"，而是"异步返回"。真正的流式应满足：

1. **模型输出增量即时透出**：模型每返回一个 token，事件立即发出。
2. **工具调用生命周期可观测**：`ToolCallStart` → `ToolCallDelta`（参数流式解析）→ `ToolCallEnd`。
3. **工具执行结果可流式消费**：长文本 tool result 可以分块透出。
4. **挂起点可中途暂停**：遇到 `RequireUserConfirmEvent` 后，事件流暂停，等待外部事件注入恢复。

#### 目标设计

```go
package agent

// Agent 接口扩展（向后兼容：新增方法，不删除旧方法）
type Agent interface {
    // 原有方法保留
    Name() string
    Call(ctx context.Context, msg *message.Msg) (*message.Msg, error)
    CallStream(ctx context.Context, msg *message.Msg) (<-chan *message.Msg, error)
    
    // V2 新增：真事件流
    ReplyStream(ctx context.Context, msg *message.Msg) (<-chan event.AgentEvent, error)
    
    // V2 新增：状态管理
    LoadState(state *agent.State) error
    SaveState() (*agent.State, error)
}

// ReActAgent 内部重构核心：
// runLoop 不再返回 *message.Msg，而是写入 eventChannel
func (a *ReActAgent) runLoop(ctx context.Context, msg *message.Msg, out chan<- event.AgentEvent) {
    out <- event.NewReplyStart(a.replyID)
    
    // 1. Reasoning: 调用模型流式接口
    for chunk := range modelStream {
        if chunk.IsThinking {
            out <- event.NewThinkingBlockDelta(...)
        } else {
            out <- event.NewTextBlockDelta(...)
        }
    }
    
    // 2. Acting: 解析 tool calls
    if hasToolCalls {
        out <- event.NewToolCallStart(...)
        
        // 3. 权限检查 → 可能产生 RequireUserConfirmEvent
        permDecision := a.permissionEngine.Evaluate(toolCalls)
        if permDecision == permission.ASK {
            out <- event.NewRequireUserConfirm(...)
            // 挂起：等待外部通过 InjectEvent() 注入 UserConfirmResultEvent
            confirmResult := a.waitForExternalEvent(ctx, confirmID)
            // ... 根据 confirmResult 过滤 tool calls
        }
        
        // 4. 执行工具
        for result := range a.executeToolsStream(ctx, toolCalls) {
            out <- event.NewToolResultTextDelta(...)
        }
    }
    
    out <- event.NewReplyEnd(a.replyID)
}
```

#### 关键技术决策

1. **Channel vs Iterator**：Go 使用 `<-chan event.AgentEvent` 天然适合事件流，比 Python 的 `AsyncGenerator` 更地道。
2. **背压处理**：消费者通过 channel 缓冲和 `context.Context` 控制消费速率；生产者（模型流式接口）通过 `ctx` 取消。
3. **挂起-恢复机制**：
   - 内部维护 `pendingExternalEvents map[string]chan event.AgentEvent`
   - `RequireUserConfirmEvent` 发出后，loop goroutine 阻塞在对应 chan 上
   - 外部调用 `agent.InjectEvent(ctx, event)` 将事件写入 chan，恢复 loop
   - 这一设计使 Agent 可以部署为无状态 HTTP handler，挂起时保存 state，恢复时加载 state

### 6.4 挂起-恢复协议（P0-4）

这是 PyV2 最具差异化的能力，也是实现 HITL 和外部执行的核心。

#### 状态流转

```
Agent 运行中
    │
    ▼
┌──────────────────────┐
│ 遇到权限规则 = ASK   │
│ 或需要外部执行        │
└──────────┬───────────┘
           ▼
    发出 RequireUserConfirmEvent
    或 RequireExternalExecutionEvent
           │
           ▼
    ┌──────────────┐
    │  SUSPENDED   │ ◄──── AgentState 保存到 Store
    └──────┬───────┘
           │
           ▼
    外部系统（人类/另一个 Agent/外部服务）
    做出决策
           │
           ▼
    注入 UserConfirmResultEvent
    或 ExternalExecutionResultEvent
           │
           ▼
    ┌──────────────┐
    │   RESUME     │ ◄──── AgentState 从 Store 恢复
    └──────┬───────┘
           │
           ▼
    Agent 继续执行
```

#### Go 实现要点

利用 Go 的 `select` + `channel` + `context` 实现优雅的挂起恢复：

```go
func (a *ReActAgent) waitForExternalEvent(ctx context.Context, confirmID string) (event.AgentEvent, error) {
    ch := a.registerWaiter(confirmID)
    defer a.unregisterWaiter(confirmID)
    
    select {
    case evt := <-ch:
        return evt, nil
    case <-ctx.Done():
        return nil, ctx.Err()
    case <-a.shutdownCh:
        return nil, ErrAgentShutdown
    }
}

// 由 HTTP handler / WebSocket handler / 另一个 goroutine 调用
func (a *ReActAgent) InjectEvent(ctx context.Context, evt event.AgentEvent) error {
    switch e := evt.(type) {
    case *event.UserConfirmResultEvent:
        ch := a.getWaiter(e.ConfirmID)
        if ch == nil {
            return fmt.Errorf("no waiter for confirm_id %s", e.ConfirmID)
        }
        select {
        case ch <- evt:
            return nil
        case <-ctx.Done():
            return ctx.Err()
        }
    // ...
    }
}
```

### 7.5 Context Compression 自动集成（P1-5）

func (a *ReActAgent) buildHistory(ctx context.Context) ([]*message.Msg, error) {
    msgs := a.memory.GetAll() // 或类似方法
    
    // 自动触发 ReMe 上下文压缩
    if preparer, ok := a.memory.(interface{ PreReasoningPrepare(context.Context, []*message.Msg) }); ok {

        }
    }
    
    return msgs, nil
}

## 附录

### A. PyV2 核心事件类型对照表

| PyV2 事件 | Go V2 事件 | 说明 |
|-----------|-----------|------|
| `ReplyStartEvent` | `ReplyStartEvent` | reply 开始 |
| `TextBlockStartEvent` | `TextBlockStartEvent` | 文本块开始 |
| `TextBlockDeltaEvent` | `TextBlockDeltaEvent` | 文本增量 |
| `TextBlockEndEvent` | `TextBlockEndEvent` | 文本块结束 |
| `ThinkingBlockStartEvent` | `ThinkingBlockStartEvent` | 思考块开始 |
| `ThinkingBlockDeltaEvent` | `ThinkingBlockDeltaEvent` | 思考增量 |
| `ThinkingBlockEndEvent` | `ThinkingBlockEndEvent` | 思考块结束 |
| `ToolCallStartEvent` | `ToolCallStartEvent` | 工具调用开始 |
| `ToolCallDeltaEvent` | `ToolCallDeltaEvent` | 工具参数增量 |
| `ToolCallEndEvent` | `ToolCallEndEvent` | 工具调用结束 |
| `RequireUserConfirmEvent` | `RequireUserConfirmEvent` | 请求用户确认 |
| `RequireExternalExecutionEvent` | `RequireExternalExecutionEvent` | 请求外部执行 |
| `ToolResultStartEvent` | `ToolResultStartEvent` | 工具结果开始 |
| `ToolResultTextDeltaEvent` | `ToolResultTextDeltaEvent` | 工具结果文本增量 |
| `ToolResultEndEvent` | `ToolResultEndEvent` | 工具结果结束 |
| `ReplyEndEvent` | `ReplyEndEvent` | reply 结束 |
| `UserConfirmResultEvent` | `UserConfirmResultEvent` | 用户确认结果（外部注入）|
| `ExternalExecutionResultEvent` | `ExternalExecutionResultEvent` | 外部执行结果（外部注入）|

### B. 术语对照

| Python v2 | Go V2 | 说明 |
|-----------|-------|------|
| `reply_stream()` | `ReplyStream()` | 真事件流 |
| `AgentState` | `state.AgentState` | 可序列化状态 |
| `PermissionEngine` | `permission.Engine` | 权限引擎 |
| `Workspace` | `workspace.Workspace` | 执行环境抽象 |
| `MiddlewareBase` | `middleware.Middleware` | 洋葱中间件 |
| `Toolkit` | `toolkit.Toolkit` | 工具集（Go 已有基础）|
| `StorageBase` | `storage.Store` | 持久化存储抽象 |

---

> **结语**：AgentScope.Go V2 的核心不是追逐功能数量，而是完成**架构范式的对齐**。在事件驱动的基础上，Go 版的 Memory 优势、并发性能和类型安全将形成与 PyV2 **互补而非替代**的差异化定位——PyV2 是 Python 生态的完整平台，Go V2 是高性能、强类型、云原生部署的首选方案。
